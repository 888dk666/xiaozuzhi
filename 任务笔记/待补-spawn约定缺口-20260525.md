2026-05-25，四轮审查（自查 + 研究文件全量对比 + 小小克外部调研 + 模拟全流程）。暂不补，等后续统一处理。

## 1. Spawn prompt 写作规范 ✅ 2026-05-25

> 六轮摸排，汇总如下。已做：ops spawn 约定加了基本必填字段表。待方案确认后补全。

### 1.0 摸排总结：十二个点

**各角色特殊场景：**
1. 多方案模式 — 三架构锚点必须不同，QA 需收齐全部方案而非单个
2. 财务时序 — 事前估算时架构方案可能还没出，"等架构出完统一给"还是"基于 PM 拆分先估"需说明
3. 工程依赖传递 — B 依赖 A 产出时，B 的 spawn prompt 必须含 A 的产出
4. QA/合规缺上下文 — QA 审"需求满足"需原始需求/成功标准，合规审"合规风险"需项目特定红线

**Spawn prompt 怎么写：**
5. 格式规范 — 结构化（`锚点: xxx / 任务: xxx`）还是自由文本？不统一则 agent 解析准确率不一致
6. PM 不要重复塞 agent 已有内容 — agent 定义文件已有角色人设和行为规则，重复塞浪费上下文
7. spawn prompt vs agent 定义文件职责边界 — PM 不知道什么该放 prompt、什么 agent 文件已有
8. prompt 不能与角色边界冲突 — PM 写"请实现"但 agent 定义是"只出方案"
9. 长度控制 — "审查内容原文"不标截断上限，50KB 贴进去子 agent 一半上下文没了

**系统约束：**
10. Spawn 后不能中途纠偏 — 子 agent 跑起来 PM 改不了 prompt，不像 Agent Teams 可以 SendMessage

**重试/失败：**
11. 重试场景 prompt — 第二次要不要跟第一次不一样？要不要附失败原因/QA 反馈？

**全局同步：**
12. 全局 CLAUDE.md + subagent-guide.md §十 还写着笼统的"任务+方案+边界"，没指向本表

社区对照：Goal + Files owned/must not touch + How to verify + 不确定性声明。

## 2. Spawn 超时/卡住判据 ✅ 2026-05-25

> 三轮摸排，汇总如下。

### 2.0 现状

spawn 约定 #7 只覆盖底层故障（网络断开/API超时/乱码），不覆盖语义层卡死。"超时"和"卡住"在 ops 升级条件和 agent prompt 里出现过，但全是定性描述——没定义多久、多少轮、什么信号。PM 无法查看子 agent 中间状态（Subagent 是阻塞调用，不像 Agent Teams 有 SendMessage），只能等结果回来或强行取消。

### 2.1 已有机制

| 机制 | 覆盖范围 | 局限性 |
|------|---------|--------|
| spawn 约定 #7 | 网络/API/乱码 | 不覆盖语义卡死 |
| 升级条件"架构超时/工程卡住" | 特定角色 | 定性描述，无阈值 |
| maxTurns（#4） | 硬机械上限 | 最后防线，早该被 #2 拦截 |
| 错误输出模板 | 记录卡死结果 | 只记录不检测 |

### 2.2 缺失项

- 统一超时阈值——5 分钟？15 分钟？不同角色需要不同阈值
- 重复输出检测——PM 看不到中间状态，只能等结果回来后人工判断
- 心跳机制——区分"agent 崩了"和"agent 还在跑"
- 统一升级阶梯——重试→扩展上下文→换方案→人工，散在各处

### 2.3 平台硬限制

- **PM 无法查看子 agent 中间状态** — Subagent 是阻塞调用，不像 Agent Teams 有 SendMessage
- **Tier 2（轻量 LLM 判断"真卡还是长任务"）无法实现** — 需要看到 agent 当前行为才能判断
- **PM 不能中途暂停问情况** — 只能等结果或取消

**结论：#2 的可行范围 = Tier 1 确定性检测 + PM 人工判断。Tier 2/3 在 Subagent 架构下不可行。**

### 2.4 交叉关联

- #4 maxTurns 是硬兜底，#2 应在其之前触发——值已按此逻辑设
- #6 错误输出模板已有错误类型分类，可复用
- #5 自检清单可加"有超时风险吗"
- 升级模板（ops §五 L1/L2/L3）已有框架
- 成本维度：没 #2 = 卡死 agent 烧到 maxTurns 才截，浪费严重

### 2.5 社区对照

三层检测：确定性（超时+重复输出+循环+心跳，免费）→ 轻量 LLM（不可行）→ 全 LLM（不可行，PM 手动替代）。生产配置：120s 心跳、5 次相同输出告警、阶梯升级。

## 4. maxTurns 未设 ✅ 2026-05-25

> 两轮摸排，汇总如下。

### 4.0 现状

五个 agent 定义文件全部零 `maxTurns`。官方：可选字段，不设 = 无上限。同时 `effort` 字段（low/medium/high/xhigh/max）也未使用——effort 控"多努力"，maxTurns 控"上限到哪"，互补但都没配。

### 4.1 关键问题

- **设多少？** 不同角色复杂度不同——架构搜+写（需多轮）、工程读+改+测（需最多轮）、QA/合规/财务只读只分析（需少轮）。统一值还是差异化？
- **截断后怎么办？** agent 达到上限被截，部分产出能不能用？PM 怎么区分"被截"和"刚好用满"？
- **静态 vs 动态** — maxTurns 是 agent 定义文件的静态字段，不能按任务复杂度动态调。简单任务和复杂任务同一个上限，设低截复杂、设高浪费简单
- **和 #2 的关系** — maxTurns 是硬机械限制（数轮次），#2 是软行为判断（看产出有效性）。先撞哪个？

### 4.2 社区对照

- CrewAI `max_iter=5`（防互踢皮球，语义不同）
- OpenAI MAX_HANDOFFS=4（防 agent 间 ping-pong，语义不同）
- LangGraph LoopAgent `max_iterations`（循环上限，类 QA 3 轮打回）
- 三者都和 Claude Code 的 maxTurns（单次 spawn 内 tool call 次数）语义不同，不能直接套用数值
- community pattern: maxTurns 作为 spawn prompt 必填字段之一

## 6. Agent 产出格式未标准化 ✅ 2026-05-25

> 五轮摸排，汇总如下。共二十点，按主题分四组。

### 6.0 现状

各角色产出格式全靠 prompt 里自然语言约定。五个 agent 文件里：架构半有（四段自由文本）、QA 有（✅/❌+原因+建议+存疑）、合规有（级别+发现+建议）、财务半有（三模式各不同）、工程**完全空白**。

### 6.1 真实翻车证据（任务笔记-20260525-0001）

唯一的真实 agent 产出样本。同一任务三个 agent 的署名格式三种写法：
- QA：`署名：QA小克 / 查理·芒格 / 晓组织 QA 总监`
- 合规：`审计官：合规小克（海瑞），晓组织合规总监`
- 财务：`署名：财务小克（本杰明·格雷厄姆）`

结论标签不一致：QA 写"结论：通过"，合规写"审计结论：通过（绿）"。合规自创了审计编号（XO-COMP-2026-05-24-001），QA/财务都没有。拼合顺序凭感觉排，呈报标题临时起名。

**这不是假设风险，是已经发生的真实翻车。**

### 6.2 拼合环节

- 拼合模板完全缺失——"拼合（不改字）"的前提是各 agent 产出能直接拼，但没有模板定义谁在前、谁在后、每段标题
- 三条路径拼合内容不同（短路径 3 份、主流程 4 份、多方案 5 份+落选方案），各自缺拼合模板
- "不改字"和"格式不一致"直接撞车

### 6.3 未定义格式的产出

- PM 自己的总结没格式
- 落选方案没格式
- 工程自验结论没格式（五个角色里被消费最多的产出，格式为零）
- 错误输出没格式（agent 报错/崩溃/超时，PM 收到原始错误信息，无法一眼判断）
- 多方案对比没格式（三架构三方案，PM 横比没有对比模板）

### 6.4 上下游消费链

- 交叉消费场景——上游格式烂了，下游一起遭殃（QA 读架构自由文本→审查质量下降；合规读工程零格式产出→巡视效率低）
- 不确定性声明只有架构有，工程产出/财务估算没有
- agent 产出之间交叉引用不完整——QA 说"审拆法 2 轮"但不引用审查产出路径
- 财务 agent 格式 vs §四 脚本不一致（agent 文件没提达标/异常/严重标签，§四 写了）

### 6.5 元数据与保存

- 产出没有统一元数据（时间戳、版本号、状态、输入引用）——合规自创审计编号是唯一例外
- 中间产出 vs 最终产出保存策略未定义
- 产出可发现性为零——未来查历史时 grep 靠运气
- 原始产出保留没机制——粘贴到任务笔记后原始来源丢失
- 产出版本追踪缺失——架构 v1 被打回→v2 通过，v1 去哪了

### 6.6 归属与强制

- 产出格式规范的归属没定——是放 agent 定义文件、ops §二、ops §四、还是独立文件？protocol.md 把这事交给了标准小克（标准化官），但标准小克还没启用
- 格式没有强制机制——定义了也没人检查。QA 审内容、合规审风险，没人审格式
- 署名结论 8 处引用 ≠ 0 次定义——到处说"署名结论"但没一个地方写它长什么样

来源：research.md §七 常见翻车——"A 输出 YAML、B 期望 JSON"。已通过真实样本验证。

## 7. 操作幂等性未声明 ✅ 2026-05-25

> 三轮摸排，汇总如下。

### 7.0 现状

全系统零次出现"幂等"。社区铁律第一条"所有操作幂等"——我们有重试/换人/升级，但从没提过幂等。

### 7.1 关键发现：只有工程受影响

五个角色中，架构/QA/合规/财务只有只读工具（Read/Grep/Glob ± WebSearch/WebFetch），天然幂等。**只有工程小克有 Write/Edit/Bash**——幂等风险完全集中在工程一人身上。

### 7.2 重试场景的幂等风险

| 场景 | 触发 | 风险 |
|------|------|------|
| L1 同 agent 重试 | 网络断开后重试 | 已写入一半，再写一遍 |
| L1 QA 打回重改 | 工程修了再交 | 同一文件改两版 |
| L2 换人接手 | 新 agent 接管 | 不知道前一个人干了什么 |
| L3 PM 自己干 | PM 接手 | 同 L2 |

### 7.3 工具差异

- Write：幂等（覆盖写）
- Edit：**不幂等**——目标字符串已被改过时匹配失败
- Bash：状态不确定——可能任何副作用

### 7.4 脏重试 vs 干净重试未区分

网络断开（agent 跑了 0 轮）和 maxTurns 截断（跑了 25 轮）被 L1 同等待遇。前者直接重试安全，后者需要先清理。

### 7.5 已有缓解措施

- worktree 隔离——工程在独立分支干活，脏了不影响主线
- 五原则 #5 "不全量原样重跑"——prompt 会变

### 7.6 社区对照

patterns.md §五 铁律第 1 条："所有操作幂等"。10+ Agent 团队的经验——重试是常态，幂等是安全前提。

## 8. SubagentStart/SubagentStop hooks ✅ 2026-05-25 已部署

> 实现方案见 [方案-gap8-实现-20260525](方案-gap8-实现-20260525.md)，摸排汇总见 [摸排汇总-gap8-20260525](摸排汇总-gap8-20260525.md)。三轮摸排，汇总如下。2026-05-25 更新：查官方文档修正 SubagentStart 不可 block 的假设。

### 8.0 现状

settings.json：SubagentStart 已注册调试 hook（抓 stdin 字段），SubagentStop 零注册。已有 16 个 hook 脚本可做模板。

注册格式：`Event → matchers[] → hooks[]（type:command + command + timeout）`。hook 接口：stdin 收 JSON → python3 处理 → stdout 输出 JSON。

**2026-05-25 官方文档验证结论：** SubagentStart **不能 block spawn**（exit 2 仅显示 stderr，不阻止 spawn）。SubagentStop **可以 block**（exit 2 阻止 agent 停止，但对计数场景无意义——我们需要的是记完成时间而非阻止停止）。完整规格见 [subagent-hooks-official-spec.md](../reference/subagent-hooks-official-spec.md)。

### 8.1 可行性（经官方文档修正）

**A. 成本日志 ✅ 完全可行：** SubagentStart 记 spawn 时间 + agent type（stdin 字段已确认，见 §8.3），SubagentStop 记完成时间。自动化 spawn 约定 #8 的成本基线追踪。

**B. 活跃数硬挡 ❌ 原方案不可行：** SubagentStart **不支持 exit 2 拒绝 spawn**（官方明确：Can block? No）。替代方案：
- SubagentStart 只做计数 + stderr 警告 + additionalContext 提醒（软提醒，不硬挡）
- 实际硬挡靠 PM 遵守 CLAUDE.md 规则（spawn 前自检清单第 4 项）或探索其他机制
- 配套：定期审查 spawn 日志，发现违规追责

### 8.2 硬限制

- **fail-open** — hook 脚本崩溃 = 静默跳过，不挡业务
- **SubagentStop 不可靠** — agent 崩溃或父会话中断时不触发。纯靠 hook 计数会漂移，需僵尸清理逻辑
- **hook 内不能调 Agent 工具** — 不能用子 agent 校验自己的 spawn。判断只能靠 state 文件 + stdin 数据
- **SubagentStart 不可 block** — 官方设计就是纯观察，不能挡 spawn。这是平台硬限制，不是实现问题

### 8.3 stdin 字段（已确认）

**SubagentStart**（调试 hook 实测 + 官方文档对照）：
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "...",
  "effort": {"level": "..."},
  "hook_event_name": "SubagentStart",
  "agent_id": "...",
  "agent_type": "Explore"
}
```

**SubagentStop**（2026-05-25 调试 hook 实测）：
```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "...",
  "permission_mode": "...",
  "effort": {"level": "..."},
  "hook_event_name": "SubagentStop",
  "agent_id": "...",
  "agent_type": "Explore",
  "agent_transcript_path": "...",
  "last_assistant_message": "子 agent 最终回复全文",
  "stop_hook_active": false,
  "background_tasks": [],
  "session_crons": []
}
```

关键发现：**`agent_transcript_path` 指向子 agent 独立转录文件，可读取提取实际 token 消耗。** 成本日志无需 PM 手动报——hook 直接从转录文件拿数据。

### 8.4 交叉关联

- #5 自检清单第 4 项（当前活跃 Agent 数）— 可由 SubagentStart hook 做**软提醒**（stderr + additionalContext），但不能硬挡。硬挡仍靠 PM 自觉
- #5 自检清单第 6 项（subagent_type 匹配）— SubagentStart 可拿到 `agent_type`，可做校验 + 软提醒
- spawn 约定 #8（成本基线）— A 部分成本日志直接对应该条
- #3 域清单 + protocol 修改必查清单 — hook 脚本注册后需同步更新 hooks-system.md 参考文档
