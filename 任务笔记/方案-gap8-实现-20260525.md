## 三、数据格式

### 3.1 成本日志 spawn-cost.jsonl

每行一条 JSON，两类记录：

**start 记录（SubagentStart 写入）：**
```json
{
  "event": "start",
  "ts": "2026-05-25T20:45:00",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore"
}
```

**stop 记录（SubagentStop 写入）：**
```json
{
  "event": "stop",
  "ts": "2026-05-25T20:45:12",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore",
  "tokens": {
    "input": 8459,
    "output": 255,
    "cache_creation": 0,
    "cache_read": 10880,
    "total": 19594
  },
  "stop_reason": "end_turn"
}
```

**stop 记录（SubagentStop 触发但转录文件读取失败时）：**
```json
{
  "event": "stop",
  "ts": "2026-05-25T20:45:12",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore",
  "tokens": null,
  "stop_reason": "unknown"
}
```

> token 总计 = input + output + cache_creation + cache_read

## 四、Hook 脚本逻辑

### 4.1 SubagentStart 处理

```
1. 读 stdin JSON
2. 提取 agent_id, agent_type, session_id
3. 记时间戳 ts
4. 追加 start 记录到 spawn-cost.jsonl
5. fcntl 锁 agent-count 状态文件
6. 僵尸清理：遍历 agents，删 started_at > 1h 的条目
7. count += 1，加 agent 条目
8. 写入状态文件，解锁
9. 构建 additionalContext（提醒子 agent 当前负载）
10. 如果 count >= 5：设 systemMessage 警告 PM
11. stdout 输出 JSON（additionalContext + systemMessage）
12. 全程异常 → exit 0（fail-open）
```

### 4.2 SubagentStop 处理

```
1. 读 stdin JSON
2. 提取 agent_id, agent_type, session_id, agent_transcript_path
3. 记时间戳 ts
4. fcntl 锁 agent-count 状态文件
5. 从 agents 中移除该 agent_id，count -= 1
6. 写入状态文件（空则删），解锁
7. 读 agent_transcript_path，提取 token 汇总 + stop_reason
8. 追加 stop 记录到 spawn-cost.jsonl
9. stdout 输出 {"continue":true}（不 block 停止）
10. 全程异常 → exit 0（fail-open）
```

### 4.3 token 提取逻辑

```
读 agent_transcript_path JSONL：
1. 遍历每条消息
2. 取 message.usage（含 input_tokens/output_tokens/cache_*）
3. 累加所有 usage 值
4. 取最后一条有 stop_reason 的消息，取其 stop_reason
5. 如果 usage 全为零或 stop_reason 为空：tokens=null, stop_reason="unknown"
```

      }
    ]

## 六、SubagentStart stdout 输出

### 正常（count < 5）

```json
{
  "continue": true,
  "hookSpecificOutput": {
    "hookEventName": "SubagentStart",
    "additionalContext": "[晓组织] 当前活跃 agent: 3/5。你是 Explore 类型，spawn 于 2026-05-25T20:45:00。"
  }
}
```

### 达到上限（count >= 5）

```json
{
  "continue": true,
  "systemMessage": "⚠ 晓组织活跃 agent 已达 5/5 上限。请确认是否继续 spawn。",
  "hookSpecificOutput": {
    "hookEventName": "SubagentStart",
    "additionalContext": "[晓组织] ⚠ 当前活跃 agent: 5/5（已达上限）。你是 Explore 类型。请高效完成任务，避免多余操作。"
  }
}
```

## 七、SubagentStop stdout 输出

```json
{"continue": true}
```

> 不做 block——SubagentStop 可 block 但我们只需要日志，不阻止 agent 停止。
> 不做 additionalContext——SubagentStop 不支持此字段。

## 八、边界处理

| 场景 | 处理 |
|------|------|
| SubagentStart stdin 字段缺失 | fail-open，仅 log 不拦截 |
| SubagentStop agent_transcript_path 为空 | tokens=null, stop_reason="unknown" |
| 转录文件不存在 | tokens=null, stop_reason="unknown" |
| 转录文件解析失败 | tokens=null, stop_reason="unknown" |
| agent-count 状态文件损坏 | 重新初始化空状态 |
| fcntl 锁超时 | 跳过计数（fail-open），成本日志不受影响 |
| spawn-cost.jsonl 写失败 | 静默跳过，不阻塞 spawn |
| 并发 spawn（多个 Start 同时） | fcntl 锁保护状态文件；JSONL 追加原子安全（每行 < PIPE_BUF） |
| 僵尸 agent（SubagentStop 从未触发） | SubagentStart 时清理 >1h 条目；SessionEnd 删整个状态文件 |
| worktree agent | 转录路径在父会话目录下，不在 worktree 内，不受清理影响 |

## 十二、已知局限（接受的风险）

| 局限 | 风险 | 缓解 |
|------|------|------|
| SubagentStart 不可 block | ≥5 个 agent 时只能警告，不能硬挡 | PM 自检清单兜底；systemMessage 明显提醒 |
| SubagentStop 崩溃时不触发 | agent 计数漂移 + 无 stop 日志 | 僵尸清理（1h TTL）+ tokens=null 记录 |
| worktree agent 未实测 | hook 行为未 100% 确认 | 待办已建；机制推理低风险 |
| 后台 agent 结果异步到达 | SubagentStop 延迟触发——agent 完成后才触发，不影响数据完整性 | 无影响 |
| 转录文件可能缺最后几行 | 实测 31KB 完整，但极端情况不可证伪 | fail-open，tokens=null 记录 |

## 十三、不做

| 项 | 原因 |
|----|------|
| PostToolUse 成本回注 | Agent 工具 stdin 不含子 agent 执行结果，**机制不通** |
| SubagentStop block（防 agent 停止） | 成本日志不需要；B 计数用减计数即可 |
| 按 agent_type 过滤 matcher | 所有类型都要追踪成本 |
| 日志轮转 | 几千条才几 MB，不需要 |
| 成本趋势自动分析 | 先积累数据，有数据再说 |
