## 三、数据格式

### 3.1 成本日志 spawn-cost.jsonl

每行一条 JSON，两类记录：

**start 记录（SubagentStart 写入）：**
```json
{
  "event": "start",
  "ts": "2026-05-25T20:45:00",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore"
}
```

**stop 记录（SubagentStop 写入）：**
```json
{
  "event": "stop",
  "ts": "2026-05-25T20:45:12",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore",
  "tokens": {
    "input": 8459,
    "output": 255,
    "cache_creation": 0,
    "cache_read": 10880,
    "total": 19594
  },
  "stop_reason": "end_turn"
}
```

**stop 记录（SubagentStop 触发但转录文件读取失败时）：**
```json
{
  "event": "stop",
  "ts": "2026-05-25T20:45:12",
  "session_id": "ae02e71c-...",
  "agent_id": "a5c75a0cae...",
  "agent_type": "Explore",
  "tokens": null,
  "stop_reason": "unknown"
}
```

> token 总计 = input + output + cache_creation + cache_read

## 四、Hook 脚本逻辑

### 4.1 SubagentStart 处理

```
1. 读 stdin JSON
2. 提取 agent_id, agent_type, session_id
3. 记时间戳 ts
4. 追加 start 记录到 spawn-cost.jsonl
5. fcntl 锁 agent-count 状态文件
6. 僵尸清理：遍历 agents，删 started_at > 1h 的条目
7. count += 1，加 agent 条目
8. 写入状态文件，解锁
9. 构建 additionalContext（提醒子 agent 当前负载）
10. 如果 count >= 5：设 systemMessage 警告 PM
11. stdout 输出 JSON（additionalContext + systemMessage）
12. 全程异常 → exit 0（fail-open）
```

### 4.2 SubagentStop 处理

```
1. 读 stdin JSON
2. 提取 agent_id, agent_type, session_id, agent_transcript_path
3. 记时间戳 ts
4. fcntl 锁 agent-count 状态文件
5. 从 agents 中移除该 agent_id，count -= 1
6. 写入状态文件（空则删），解锁
7. 读 agent_transcript_path，提取 token 汇总 + stop_reason
8. 追加 stop 记录到 spawn-cost.jsonl
9. stdout 输出 {"continue":true}（不 block 停止）
10. 全程异常 → exit 0（fail-open）
```

### 4.3 token 提取逻辑

从 `agent_transcript_path` JSONL 文件中提取：

```
tokens = {input: 0, output: 0, cache_creation: 0, cache_read: 0}
stop_reason = null

for each line in agent_transcript_path:
    msg = parse_json(line)
    if msg.message.role != "assistant":
        continue
    usage = msg.message.usage
    if usage:
        tokens.input += usage.input_tokens || 0
        tokens.output += usage.output_tokens || 0
        tokens.cache_creation += usage.cache_creation_input_tokens || 0
        tokens.cache_read += usage.cache_read_input_tokens || 0
    sr = msg.message.stop_reason
    if sr and sr != "tool_use":  # tool_use 是中间消息，取后续的
        stop_reason = sr

tokens.total = tokens.input + tokens.output + tokens.cache_creation + tokens.cache_read

if tokens.total == 0:
    tokens = null
    stop_reason = "unknown"
```

> 实测验证：Assistant 消息的 usage 对象含所有 4 个字段。`tool_use` stop_reason → 忽略该条，取后续消息的 stop_reason。最终 `end_turn` / `max_tokens` / `stop_sequence` / `refusal` / `pause_turn` 之一。

### 4.4 JSONL 追加安全

每行 JSON < 500 字节，远小于 macOS PIPE_BUF（4096）。POSIX 保证 `write()` 在 PIPE_BUF 内原子执行。多个 SubagentStart/SubagentStop 并发追加 spawn-cost.jsonl **无需加锁**——每行要么完整写入，要么不写入（hook 崩溃时）。

### 4.5 状态文件锁

```
读 agent_transcript_path JSONL：
1. 遍历每条消息
2. 取 message.usage（含 input_tokens/output_tokens/cache_*）
3. 累加所有 usage 值
4. 取最后一条有 stop_reason 的消息，取其 stop_reason
5. 如果 usage 全为零或 stop_reason 为空：tokens=null, stop_reason="unknown"
```

## 六、SubagentStart stdout 输出

### 正常（count < 5）

```json

  "continue": true,
  "hookSpecificOutput": {
    "hookEventName": "SubagentStart",
    "additionalContext": "[晓组织] 当前活跃 agent: 3/5。你是 Explore 类型，spawn 于 2026-05-25T20:45:00。"

```

### 达到上限（count >= 5）

```json
{
  "continue": true,
  "systemMessage": "⚠ 晓组织活跃 agent 已达 5/5 上限。请确认是否继续 spawn。",
  "hookSpecificOutput": {
    "hookEventName": "SubagentStart",
    "additionalContext": "[晓组织] ⚠ 当前活跃 agent: 5/5（已达上限）。你是 Explore 类型。请高效完成任务，避免多余操作。"
  }
}
```

## 七、SubagentStop stdout 输出

```json
{"continue": true}
```

> 不做 block——SubagentStop 可 block 但我们只需要日志，不阻止 agent 停止。
> 不做 additionalContext——SubagentStop 不支持此字段。

## 八、边界处理

| 场景 | 处理 |
|------|------|
| SubagentStart stdin 字段缺失 | fail-open，仅 log 不拦截 |
| SubagentStop agent_transcript_path 为空 | tokens=null, stop_reason="unknown" |
| 转录文件不存在 | tokens=null, stop_reason="unknown" |
| 转录文件解析失败 | tokens=null, stop_reason="unknown" |
| agent-count 状态文件损坏 | 重新初始化空状态 |
| fcntl 锁超时 | 跳过计数（fail-open），成本日志不受影响 |
| spawn-cost.jsonl 写失败 | 静默跳过，不阻塞 spawn |
| 并发 spawn（多个 Start 同时） | fcntl 锁保护状态文件；JSONL 追加原子安全（每行 < PIPE_BUF） |
| 僵尸 agent（SubagentStop 从未触发） | SubagentStart 时清理 >1h 条目 |
| worktree agent | 转录路径在父会话目录下，不在 worktree 内，不受清理影响 |

## 九、跨会话安全

### 9.1 状态文件隔离

状态文件含 session_id：`/tmp/agent-count-{session_id}.json`。不同会话不会互相覆盖。

### 9.2 僵尸清理策略

**不依赖 SessionEnd 清理。** 原因：`rm -f /tmp/agent-count-*.json` 会误删其他并行会话的状态文件（跨会话竞态）。
- 唯一清理路径：SubagentStart 时扫描本会话状态文件，去除 started_at > 1h 的僵尸条目
- 死会话残留文件（SessionEnd 未触发）留在 /tmp/，单文件 ~200B，可忽略
- SessionStart 不做主动清理——不存在跨会话计数需求
