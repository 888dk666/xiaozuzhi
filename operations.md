## 一、Spawn 命令

PM小克（当前 Claude 实例）收到 CEO 指令后，按以下规则逐一派发 Agent。

**标注：** `→` 必须等上一步完成 | `∥` 可以同时派发

**每个步骤 = 一次 Agent 工具调用。** PM小克负责：按"Spawn prompt 必填字段"表准备 prompt → 调用 Agent 工具 → 接收结果 → 决定下一步。角色 system prompt 由 agent 定义文件自动加载（§二 为注释版供人阅读参考）。

**Spawn 约定** — 通用机制见 [小小克使用指南](~/projects/小小克/subagent-guide.md)，调研依据见 [research-spawn.md](research-spawn.md)，角色 agent 定义文件见 [.claude/agents/](.claude/agents/)，团队上限依据见 [protocol.md §七](protocol.md)。

1. 上下文：不传对话历史，只传任务特定信息。各角色必填字段见下方"Spawn prompt 必填字段"表
2. 模型：PM=Opus，架构/QA=Sonnet，合规/财务=Sonnet，工程=继承PM模型（`inherit`，不硬绑Haiku——搜索/调研类任务需要强推理，简单代码修改成本可控）。Coordinator Mode：暂不开，等 10 次任务后评估
3. 权限模式 + isolation：按角色分配，见 agent 定义文件
4. 团队：同时活跃上限 5，见 protocol.md §七
5. 不内置：Agent Teams / Fork 模式 / Agent-based Hook
6. PM 自己干 vs 派 Agent 判据：只读/纯思考 → PM 自己干；需写文件/搜索/执行 → 派 Agent
7. 失败处理：网络断开→重试 2 次后报 PM；API 超时→降级用 Haiku；子 Agent 乱码→换人重跑。异常终止按错误输出模板记录处置决策（模板见下方拼合模板区）。以上为首次故障响应，持续失败按下方升级阶梯逐级上报。background agent 完全不可见，无超时阈值——仅靠 maxTurns 兜底
8. 成本基线：首次完整任务（2026-05-25，钱学森研究学习）实测 ~7.6M tokens / 9 spawns（研究型任务，含 3 轮 QA）。简单代码任务预计显著低于此值。后续任务继续采集
9. 操作幂等——重试前告知 agent 这是重试 + 上次做到哪一步（worktree 隔离已保证每次环境干净）
10. **Agent 文件部署必须用硬链接，不能用 symlink**——Claude Code 扫描 agent 目录时不跟随符号链接。已验证：PUA 插件 agent（普通文件）可被发现，symlink agent 全部不可用。部署方式：`python3 -c "import os; os.link(src, dst)"`（同 APFS 卷可行）。**部署后必须验证**：重启会话 → spawn 一次任意角色 → 确认出现在可用 agent 列表中。不验 = 没部署。**Edit 工具会打断硬链接**（写新文件+原子替换→新 inode），每次 Edit agent 文件后必须重建。事故来源：2026-05-25 晓组织首次实战，五个 agent symlink 全部不可用，白耗 3 轮 QA 改用 general-purpose 绕行

**超时参考（PM 心理锚点，非系统自动截断）：**

| 角色 | 参考超时 | 超时后动作 |
|------|---------|-----------|
| 架构小克 | 10min | PM 检查：完成了？在推进？→ 继续等 / 干预 |
| 工程小克 | 10min | 同上 |
| QA小克 | 5min | 同上 |
| 合规小克 | 5min | 同上 |
| 财务小克 | 5min | 同上 |

Subagent 默认是阻塞调用（前台），PM 无法查看中间状态。后台模式（Agent 工具传 `background: true` 或子代理定义设 `background: true` frontmatter）可并发运行，但权限提示自动拒绝。晓组织当前只用前台模式——PM 逐个 spawn、逐个验收。超时不是自动截断——是 PM 决定"该去看一眼了"的心理锚点。#4 maxTurns 是硬兜底——在卡死被 PM 发现之前，至少不会让 agent 无限烧 token（每次卡死 = 浪费完整 maxTurns 预算）。

**轮次预算（2026-05-28 补）：** ITBench-AA 基准实测——GPT-5.5 用 31 轮拿 46%，Gemini 3.1 用 83 轮只拿 30%。轮次多 2.7 倍、得分低 16 个百分点。根因：过度调查→挖出"原因的原因"当根因→误报。**更长≠更好。** 工程师 agent 按任务复杂度设 maxTurns：简单任务 ≤30、中等 ≤50、复杂任务分两次 spawn（阶段一 ≤50 轮 + 阶段二继续），不在单次 spawn 里硬撑。engineer agent 定义 maxTurns=50 为硬上限。

**疑似卡死三特征（PM 收到结果后逐条对照）：**
- 结果为空 / 被截断（非 maxTurns 截断）
- 产出与任务完全无关
- 与上次产出几乎一样（跨 spawn 比对）

中一条 → 走升级阶梯 L1。

**升级阶梯（持续失败时逐级上报）：**

| 级 | 触发 | 动作 | 对应现有机制 |
|----|------|------|------------|
| L1 | 首次失败 | 重试（同配置，附反馈），最多 2 次 | spawn 约定 #7 |
| L2 | L1 用尽仍失败 | 换方案（改 prompt / 换 agent 类型 / 降级模型） | 架构→纯牛马、API→Haiku |
| L3 | L2 仍不行 | PM 自己干 | 升级条件 |
| L4 | PM 也干不了 | 报 DK666 | 升级模板 L3 |

> 注：本阶梯适用于 spawn 故障升级。Agent 间意见分歧另有升级模板（见 ops §五 L1/L2/L3），层级同名但触发条件不同，注意区分。

> **已知限制：** Subagent 是阻塞调用，Tier 2（LLM 判断"真卡还是长任务"）和 Tier 3（LLM 出纠正指令）在此架构下不可行。PM 事后判断替代。Agent Teams 的 SendMessage 中途询问能力在此不可用。

**Spawn prompt 必填字段** — PM 派 Agent 前逐项核对，缺一不可：

| 角色 | 必填字段 | 特殊说明 |
|------|---------|---------|
| 架构小克 | 锚点 + 任务描述 | 多方案时三个架构锚点必须不同 |
| QA小克 | 审查模式 + 被审对象 + 审查内容原文 + 原始需求/成功标准 + 威胁模型 | 多方案时需收齐全部方案；原文建议 ≤5000 字；威胁模型见 Step 0.2-6，PM 未定义→QA 标注"威胁模型缺失" |
| 合规小克 | 方案摘要 + 执行结果 + 项目特定红线 | 红线引用全局 CLAUDE.md 或晓组织 CLAUDE.md |
| 财务小克 | 模式 + 方案或实际消耗 | 事前估算：等架构出完统一给；如并行则基于 PM 拆分先估 |
| 纯牛马 | 任务 + 方案(如有) + 边界 | B 依赖 A 产出时，spawn B 必含 A 的产出 |

**Spawn prompt 写作五原则：**

1. **区分职责** — spawn prompt 放任务特定信息（干什么、上下文、约束）；agent 定义文件已有角色通用规则（怎么干、边界），不重复。
2. **格式统一** — 用标签格式：`锚点: xxx` `任务: xxx` `边界: xxx`。不混用自由文本。
3. **检查边界** — spawn 前确认 prompt 与角色定义一致（不写"请实现"给架构、不写"出方案"给工程）。
4. **一次性** — spawn 后不能改 prompt，这是全部。确认清楚再派。
5. **重试必附反馈** — 第二次 spawn 必须附失败原因/QA 反馈/上次做到哪一步/与前次差异，不全量原样重跑。

**Spawn 前自检清单** — PM 每次 spawn 前逐项打勾：

基本（每次必过）：
| # | 检查项 | 依据 |
|---|--------|------|
| 1 | 必填字段齐了吗？ | 上方必填字段表 |
| 2 | prompt 和角色边界一致吗？ | 五原则 #3 |
| 3 | prompt 最终确认过了？（spawn 后不能改） | 五原则 #4 |
| 4 | 当前活跃 Agent 数 < 5？ | spawn 约定 #4 |
| 5 | 模型选对了吗？ | spawn 约定 #2 + agent 定义文件 |
| 6 | subagent_type 和角色匹配吗？model 没意外覆盖 agent 定义？ | agent 定义文件 |
| 7 | 收到结果后下一步知道吗？ | 衔接计划 |

复杂任务首次 spawn 前加过：
| # | 检查项 | 依据 |
|---|--------|------|
| 8 | 我真的需要 spawn 吗？ | spawn 约定 #6 |
| 9 | 大概花多少 token？ | spawn 约定 #8 |
| 10 | 失败后衔接方案有吗？（谁重试/谁换人，含幂等——本次是重试吗？上次做到哪一步？） | spawn 约定 #7 |
| 11 | git 状态干净？worktree 无冲突？ | 工程 agent 配了 isolation |
| 12 | 工程 spawn 时有验证命令/通过标准吗？ | #1 社区对照 |
| 13 | 产出格式 agent 知道吗？ | agent 定义文件有模板 |
| 14 | 这个任务有超时风险吗？角色超时参考是多少？ | spawn 约定超时参考 |
| 15 | 威胁模型给 QA 了吗？（Step 0.2-6 产物，QA 以此为优先级判据） | spawn 必填字段 QA行 |

### 步骤 0：任务分类（路由开关）【所有路径共用】

PM 接到 CEO 需求后，先分类再动手。不派 Agent，PM 自己完成。

```
0.1 查历史 + 了解现状

    0.1.0 提取关键词 — 从 CEO 指令中提取 2-3 个核心词 + 1-2 个功能词
        例："LangGraph"+"CrewAI"+"框架对比"+"多Agent"
        这些关键词用于下面所有搜索

    0.1.1 了解现状 — 任务要碰的是什么？

        做什么：搞清楚四件事 —
        1. 涉及哪些文件/模块？（用 0.1.0 关键词 grep 定位）
        2. 这些文件现在什么样？（Read 关键文件，快速浏览结构）
        3. 有什么关联依赖？（import/require/引用，别改了 A 炸了 B）
        4. 有什么硬约束？（CLAUDE.md 红线、现有 hook/guard、配置文件限制）

        怎么做：三步操作，最多 3 分钟 —
        a. grep 关键词搜索 → 找到相关文件路径
        b. Read 定位到的关键文件（选最核心的 1-2 个，读结构不逐行）
        c. 对照 CLAUDE.md 检查红线/约束

        产出：写入任务笔记 —
        "涉及文件：<列表> | 关键约束：<红线/依赖/hook> | 注意：<边界>"

        范围控制 —
        只搜直接相关的。不确定范围 → grep 先搜，看结果再收窄
        这是侦察，不是代码审查——不深读逻辑，不追调用链

    0.1.2 查历史 — 这事之前做过类似的吗？──────────
    ├── project/records.md       → grep/Read 按关键词搜过往同类任务
    ├── incident/log.md          → grep/Read 按关键词搜踩坑记录
    ├── feedback/rules.md        → grep/Read 搜相关行为约束
    ├── CLAUDE.md                → 已在上下文，检查红线
    └── 晓组织任务归档           → grep（跑过任务后），搜上次同类任务

    0.1.3 查到后怎么做 ────────────────────────────
    同类任务记录 → 提取上次路径、拆法、工程数 → 对照差异，不照抄
    踩坑记录     → 记"⚠️ 上次踩过 X，这次注意 Y"
    行为约束     → 标注"本次约束：X"，执行时自动遵守
    红线风险     → 可能碰→标"⚠️ 红线：X"；不确定也标，后续决定是否问 CEO
    QA 结论      → 上次常出什么缺陷 → 预判这次
    财务 ROI     → 上次 cost baseline → 这次估算参考

    0.1.4 写入任务笔记 ────────────────────────────
    路径：~/projects/晓组织/任务笔记/（持久化，跨会话不丢）
    文件名：任务笔记-{YYYYMMDD}-{序号}.md（走晓组织流程的任务）
           审查笔记-{YYYYMMDD}-{序号}.md（审阅/讨论/发现，未跑晓组织流程）
    以上所有发现汇总写入"## 0.1 查历史+现状"段
    现状：<0.1.1 产出>
    历史：<0.1.2+0.1.3 产出>
    没查到 → 写"未查到类似任务，本次为首次"
    这一步的唯一产出是笔记——没写 = 没落地。任务笔记区域说明（原始产出/废弃版本/过程记录/呈报）见 §一 拼合模板区。末尾加一行 OODA 状态：Observe=[已查到X]，Orient=[这意味着Y]，Decide=[因此应走Z路径]

0.2 理解需求 — 四件事
    0. 先读任务笔记 0.1 段 — 把历史发现（已有报告/踩坑/约束/红线）注入理解过程
    1. 读 CEO 指令 + 0.1 结论 → 写复述："CEO 要的是 X，目的是 Y"
       目的不清楚？→ 问 CEO（晓组织正常档）/ 合理假设并标注"假设的目的"（晓组织静音档，即卡卡西）
    2. 定成功标准 — "怎样算完成了？达到什么程度？深度？格式？"
       0.1 发现直接影响成功标准：已有报告 → 标准不是"重新调研"，是"基于已有数据出推荐"
    3. 复述或标准拿不准 → 问 CEO 确认（晓组织正常档）/"假设"标注（晓组织静音档，即卡卡西）
    4. 研究任务定成功标准时，将"研究生水平"操作化为 4 条可验证标准：
       ① 论证重构 — 能用自己的话独立重构核心论证链（从前提→结论）
       ② 批评理解 — 能对比至少 2 个主要批评者，指出其论证的强项和弱项
       ③ 跨学派对比 — 能对比至少 2 个相关学派在该问题上的立场差异
       ④ 源追溯链 — 核心论断能追溯到具体的 L1/L2 一手来源
    5. 可选自问：这个课题对 DK666 的实际工作有多少可迁移价值？PM 自答标注到任务笔记（卡卡西模式下标注但不自行叫停）
    6. 定威胁模型 — "出什么问题我们真的在乎？"
       不是列举所有可能的风险。是定义本次任务的"值得修"边界——QA 审查时以此为判据。
       写 2-3 条：什么偏差会直接导致任务失败？什么问题"知道了但不影响交付"？
       在此边界外的问题→标注"已知缺口"，不阻塞交付。

0.2.5 判任务类型 — 决定后续所有路由判断的基准
    必须在 0.3 拆任务之前完成——任务类型错了，拆法、路径、角色分配全错。

    问一句：这个任务核心操作是什么？
    ├── 写代码/改配置/部署/改文件 → 代码类
    ├── 搜索/调研/学习/阅读/分析/综合 → 研究类
    └── 两者都有 → 混合类

    写入任务笔记："## 0.2.5 任务类型：{代码类/研究类/混合类}"

    判据边界：
    - 哪怕最终产出是文档，只要核心操作是搜索+阅读+综合 → 研究类
    - 混合类按"哪部分更核心"决定主导类型，另一部分作为子任务
    - 拿不准 → 默认代码类（保守，当前角色体系主要为代码类设计）

    任务类型直接影响：
    - 0.3 拆法（研究类不需要"可能影响文件表"）
    - 0.4 路径选择（研究类有自己的路由）
    - 角色分配（研究类跳过架构师/合规）

0.3 脑中快速拆 — 八步
    0. 先读任务笔记 0.2.5 段 — 明确了任务类型再拆
    1. 先读任务笔记 0.1+0.2 段 — 历史发现 + 复述 + 成功标准

    2. 侦察搜索 — PM 对这个领域够熟悉吗？
        ├── 熟悉（历史有类似任务，或 PM 已理解）→ 跳过，直接 0.3.3
        └── 不熟悉（陌生领域，无法拆操作）→ 快速搜一次
            工具：WebSearch 一次 或 Read 一个关键文件
            目的：建立最基础领域理解，够拆操作就行
            不是完整搜索——完整搜索归纯牛马
            产出：3-5 行"侦察发现"写入笔记 0.3 段
            限 1 次操作，不深挖

    3. 引用 0.1 发现 — 历史结论直接影响操作拆解
       操作：读任务笔记 0.1 段 → 提取结论
       已有报告 → 写"基于已有数据，不重新搜网络"
       无历史 → 写"从零调研"

    4. 列操作 — 对照 0.2 的成功标准定每步深度
       操作：读任务笔记 0.2 成功标准 → 套对照表 →
       成功标准含"深度/全面/对比" → 至少 2 轮分析（初析+交叉验证）
       成功标准含"快速/概要/一句话" → 1 轮分析即可
       拿不准或介于两者之间 → 默认 2 轮（保守，宁可多分析不欠深度）
       每个操作写出：类型（读/写/搜索/改代码）、多深、多重

    5. 多路径探测 — 这件事存在 ≥2 个合理但不同的做法吗？
       操作：PM 自问 → 写进笔记
        ├── 有 → 标注"方向不确定"，0.4 走多方案竞标
        ├── 没有 → 默认单路径，不硬造多方案
        └── 不确定 → 标注"存疑"，走主流程单方案（保守）
    6. 需要外部信息吗？
       ├── 不需要 → 跳过，写"无外部信息依赖"
       └── 需要 → IQF 检查三步 ─────────────────
           a. 列出所有外部信息源 — 搜什么、读什么 URL、参考什么文档
           b. 逐源定权重：
               L1 官方文档/源码 → 可信，直接用
               L2 官方媒体/大学出版 → 可信，标注
               L3 百科/新闻 → 可用，不唯一引用
               L4 自媒体(知乎/简书/CSDN) → 需交叉验证
               排除 AI 生成内容 → 不引用
           c. 标注风险到任务笔记：
               "⚠️ 信息源：知乎(L4)→需交叉验证"
               "⛔ AI污染：某文标注AI生成→排除"
    7. 操作互相依赖吗？→ 哪些可以并行，哪些必须串行
    8. 复杂度自评 — PM 三项打分 ──────────────────
       操作类型：纯读+写=简单 / 含搜索=中等 / 含改代码或改配置=复杂
       外部依赖：零(本地够)=简单 / 需搜索但源可靠=中等 / 多源交叉或网络不稳=复杂
       风险面：不改文件或只加新文件=简单 / 改已有代码逻辑=中等 / 改hook/guard/防线/密钥=复杂

       三项全简单 → 简单
       任一项复杂 → 复杂
       其余 → 中等

       裁决：简单 + ≤4步 → 短路径正常
             中等/复杂 → 保守走主流程（防低估）

0.4 套三条件判断路径 —
    0. 先读任务笔记 0.3 段 — 脑中拆的结论（步数、多路径、复杂度、任务类型）
    0.5 任务类型路由 — 优先于三条件 ────────
        ├── 研究类 → 跳转 研究路径。三条件不适用于研究任务——搜索+阅读+综合天然 >4步且不需要技术方案
        ├── 混合类 → 分拆：研究部分走研究路径，代码部分走三条件判断。PM 协调两者衔接
        └── 代码类 → 继续下面的三条件判断
    0.4.1 微路径前置判据 — 满足以下全部条件 → 走微路径（PM 自己做，不 spawn，不 QA 审，任务笔记写一段即可）：
        ├── ≤2 步 + PM 自己 5 分钟内能干完
        ├── 不改代码逻辑（只改文档/注释/说明性内容）
        ├── 不碰 hook/guard/防线/密钥/CLAUDE.md
        └── 不满足 → 继续三条件判断
    ① 总共 ≤4 步？（≤4 + 简单=短路径；≤4 + 中等/复杂=主流程；>4=主流程）
    ② 不需要选方案？（来自 0.3.5，≥2 可行路径=需要选=走多方案竞标）
    ③ 一个工程能独立干完？（来自 0.3.7 依赖 + 0.3.8 复杂度）
    ├── 三全满足 + 复杂度简单 → 短路径
    ├── 任一不满足 → 主流程
    │      └── 方向不确定（0.3.5=≥2路径）→ 走多方案竞标
    └── 拿不准 → 默认主流程（宁可多走，不冒走错的风险）

0.5 定路径
    1. 写路径决定到任务笔记："## 0.5 路径：{研究路径/短路径/主流程方向确定/主流程方向不确定}"
    2. 检查笔记完整性 → 0.1 + 0.2 + 0.2.5 + 0.3 + 0.4 + 0.5 六段都有内容？
    2.5 检查红线标记 — 任务笔记 0.1 段有"⚠️ 红线"标记吗？
        ├── 有 且 未跟 CEO 确认过 → 暂停，问 CEO："这个任务涉及红线 X，要继续吗？"
        │     CEO 放行 → 继续
        │     CEO 叫停 → 任务终止
        └── 无 或 已确认 → 继续
    2.5.5 首次机制风险检查 — 检查是否有"首次使用的机制"（标注"实验性"的操作）。
       如有→首次使用 + 卡卡西模式 = ⚠️ 高风险组合。建议升级为正常档（中间关键节点报 CEO），至少首次跑通后再切卡卡西。
    2.6 交叉校验 — 六段之间有没有矛盾？
        对照三组关键配对：
        a. 0.2.5 任务类型 vs 0.4 路径选择 → 研究类被判进主流程？矛盾 → 回 0.4 重判（这是最容易犯的错误——忘了先判任务类型）
        b. 0.2 成功标准 vs 0.3 复杂度自评 → "深度分析+横向对比"配"简单"？矛盾 → 回 0.3 重评
        c. 0.3 步数 vs 0.4 路径选择 → ≤4 步判了主流程或 >4 步判了短路径？矛盾 → 回 0.4 重判
        无矛盾 → 继续
        有矛盾 → 修正后再跳转，不带着矛盾进路径
    3. 完整且无矛盾 → 跳转到对应路径的步骤 1
    4. 不完整 → 补上缺失段，再跳转
    5. 进入路径后，首次 spawn 前对照 spawn 前自检清单过一遍

### 研究路径（搜索/调研/学习任务）

**触发条件：** 0.2.5 判为研究类——核心操作是搜索、阅读、分析、综合，不涉及写代码或改配置。

**适用范围：** 本路径用于"系统化学习已有知识"，不适用于创造新知识。适用：需要快速达到某个已有领域的较高理解水平。不适用：发现新知识、提出新理论、原创性学术工作。

**与短路径/主流程的关键区别：**
- 不需要技术方案 → 跳过架构小克
- 不涉及代码/安全/权限 → 合规小克可豁免（但 PM 在步骤 5 拼合时必须自检 AI 污染判据——有署名+有机构+有参考文献）
- 搜索成本可预测 → 财务小克轻量成本摘要（基于 spawn-cost.jsonl 实际消耗出具一句话，不做估算对比）
- 认知加工（理解、综合、判断）→ PM 自己做，不分配给纯牛马

**两端有 QA，中间 PM 自己学：** 搜索阶段有 QA 审搜索质量（步骤4），学习阶段有 QA 审文档过程质量（步骤7）。PM 自己做认知加工（步骤6），但产出要过 QA 的 6 项过程审查——不是审内容对不对，是审文档是否完整、自检是否诚实、引用是否可追溯。详见 [晓组织首次实战 memory](~/.claude/projects/-Users-dkhzymini/memory/project/xiaozuzhi-first-combat.md)。

```
步骤0：PM 侦察（可选）    │  如果 PM 对领域不熟悉，快速搜一次建立基础理解。
    │  限 1 次搜索操作，够拆任务就行。完整搜索归纯牛马。

    │  产出：拆分好的搜索子任务 + 每路的搜索关键词和信息源优先级。

步骤1.5：PM 自审拆法（不 spawn，PM 自己 3 分钟过一遍）
    │  对照 0.2 成功标准自问：步骤编号连续？每步有输入输出？依赖关系标注？失败路径有兜底？
    │  预检最可能被打回的 3 处：我最不确定的是什么？这 3 处处理了吗？
    │  自审通过→交 QA。自审发现明显问题→先修再交。

步骤2：QA 审拆法          │  审查搜索维度是否覆盖完整、信息源层次是否合理。
    │  打回 → PM 重拆，最多 3 轮。

步骤3：[并行] 试点 + 三路搜索    │  先试点一路验证信息源可获取 → 达标后再并行展开。
    │  试点返回后 PM 自问：试点结果是否暴露了维度划分的盲区？是否需要调整 A/B/C 路定义？如需调整→回步骤 1 重拆→步骤 2 QA 再审。
    │  Agent: 纯牛马 × N（每路一个搜索维度）
    │  每路产出写入 ~/projects/晓组织/任务笔记/搜索报告-{任务名}-{路名}-{日期}.md
    │  PM spawn 时必须在 prompt 中指定搜索报告格式（engineer 默认格式为代码导向，不适用）：
    │    - 搜索策略：使用的关键词、信息源、筛选标准
    │    - 源清单：每条含 URL/来源类型(L1-L4)/全文可获取性/AI污染判据/内容摘要
    │    - 对每个 L1 源标注：关键段落原文引用（至少 2 段），不只是摘要
    │    - 关键发现：3-5 条本路最重要的发现
    │    - 未获取项：搜索了但没拿到全文的重要源

步骤4：QA 审搜索结果      │  审查维度：覆盖度/信息源层次/去重/可获取性。
    │  PM spawn QA 时须在 prompt 中附搜索报告文件路径，供 QA 对照原文验证。
    │  打回 → 对应纯牛马重搜，最多 3 轮。

步骤5：PM 拼合搜索报告    │  先执行三路去重——重叠源标注"多路覆盖"，信息冲突处标注"待交叉验证"。
    │  PM 自检：三路返回时间与源量是否匹配？某路源量显著偏多但返回偏快→标注"⚠️ 质量存疑"，拼合时降权。
    │  PM 自检 AI 污染：54 源是否全部通过 AI 污染判据（有署名+有机构+有参考文献）？未通过的→标注"⚠️ 疑似AI生成"并排除。
    │  筛选核心源时记录筛选逻辑——入选标准（L1优先/直接相关/全文可获取）、排除原因（付费墙/质量不足/重复）。
    │  整合为完整资料集。PM 在步骤 6 学习转化前，至少直接阅读 3 篇 L1 核心源的关键段落原文。

步骤6：PM 学习转化（两阶段——先验证真懂，再写文档）
    │
    │  6a. 闭卷默写（retrieval practice + calibration）
    │      关掉所有源——包括 Agent 返回的搜索报告、对话上下文中的摘要。
    │      从记忆中提取，不从上下文中复述。
    │      默写每个核心概念的定义、关键论证、与其他概念的关系。
    │      每条标自信度：[确定]=能独立重构完整论证 / [不确定]=记得概念但无法完整重构 / [推断]=基于已有知识的合理推测未经验证。
    │      默写完成后才对照原文校准——标了"确定"但错了→针对性重学。这是 DKE 防线。
    │      建议 6a 与 6c（写文档）之间 ≥30 分钟间隔。无法间隔时标注"⚠️ 短期记忆污染风险"。
    │
    │  6b. 教新手（protégé effect）
    │      假设教一个完全不懂的人。每概念 ≤200 字讲清楚。
    │      规则：不准术语不解释、不准引用原文。自己听一遍，标出含糊处→没真懂。
    │      产出写入学习成果文档附录——"费曼教学输出"。
    │
    │  6c. 输出学习文档 — 基于 6a+6b 发现的真懂和漏洞，写正式文档
    │      PM 写完初稿后，至少间隔 10 分钟再自审一轮。自审发现的修正处标注 [修订]。
    │      文档末尾必须有"学习困惑"节——记录反复读但不完全理解的点、看似自相矛盾的张力。无困惑=学习不够深。
    │  6d. 四项自检 — 对照研究生水平自检标准逐项自评（含 6a 校准结果）
    │
    │  产出：学习文档 + 自检结果

步骤7：QA 审学习成果（两阶段——先审文档+出题，再判回答）
    │  Agent: QA小克（两次 spawn——subagent 不能交互式等待）
    │  输入：学习文档 + 自检结果（含 6a 自信度标注）+ 原始成功标准
    │
    │  7a. 文档审查 + 出题（spawn 1，标注"口试模式：出题"）
    │      QA 完成 7+1 项文档审查（见下方清单）+ 生成 5 个口试追问（遵循 7 条设计原则）
    │      文档审查打回 → PM 修改，最多 3 轮。
    │
    │  PM 回答（主会话，不 spawn）— PM 拿到 5 个问题，闭卷回答（不准查任何资料，包括自己刚写的学习文档）。
    │      建议口试与文档写作之间 ≥30 分钟间隔。无法间隔时标注"⚠️ 短期记忆污染"。
    │
    │  7b. 口试判定 + 署名结论（spawn 2，标注"口试模式：判定"，合并步骤8）
    │      QA 拿到 PM 的闭卷回答→逐题判：✅真懂 / ⚠️部分懂 / ❌没懂→写署名结论
    │      部分懂→PM 针对性补学→更新文档相关节→QA 只重审该节。
    │      没懂→打回，PM 重学→重交文档→重口试（最多 3 轮）。
    │
    │  异常处理：
    │      - QA 口试超时/失败→降级为纯文档审查（仅 7a），不阻塞交付
    │      - 轻量学习任务（≤2 个核心概念）→任务笔记标"免口试"，6a+6b 仍执行
    │      - 口试实验设计（1/3 窗口）：
    │        成功标准：口试能探到 PM 至少 1 处知识边界（不是 5/5 全对）
    │          AND QA 判定在留存测试中不被推翻
    │        失败标准：3 次全部 5/5 ✅ 且留存测试显示显著遗忘→题目难度偏低
    │        测量：每次口试后记录题目难度分布 + ✅/⚠️/❌ 比例 + PM 自评 vs QA 判定差异
    │        3 次实验窗口用尽→PM 必须在当次任务笔记中写"口试机制评估——保留/调整/放弃"三选一+理由。不评估=未闭合。

步骤8：[并行] QA 署名结论 + 财务成本摘要    │  口试任务中步骤8已合并入7b（口试判定+署名结论同一次 spawn）。
    │  无口试任务中 Agent: QA小克 写审查结论（署名，基于步骤2审拆法+步骤4审搜索+步骤7审学习成果）。
    ∥ 财务小克轻量成本摘要（基于 spawn-cost.jsonl 实际消耗出具一句话，积累研究任务成本基线）
    │
    │  PM 收齐 → 按拼合模板拼合 → 追加到任务笔记 → 呈报 CEO

步骤9（默认执行）：留存测试
    │  任务完成后 ≥1 天，PM 重答口试 5 题中的 2 道核心题。
    │  能答出完整论证→通过。答不出→❌ 留存失败，课题降级为"需复习"。
    │  不执行须在任务笔记标注豁免原因。

Agent 调用次数：5-11 次
```

**学习成果审查清单（步骤7 QA 用）：**

QA 审学习成果时，不审"内容对不对"，审"过程到不到位"：

| # | 审查项 | 标准 |
|---|--------|------|
| 1 | 结构完整 | 成功标准要求的维度都有对应章节？缺哪个？ |
| 2 | 引用可追溯 | 核心论断能追溯到具体来源？"据搜索报告"类引用是否标注了验证状态？ |
| 3 | 自检诚实 | 研究生水平 N 项标准逐项自评了吗？自评是"见第X章"有实质内容，还是走过场？含 6a 闭卷默写校准结果？ |
| 4 | 内部一致 | 文档前后有无逻辑矛盾？（正面论述与批评章节打架、同一数字两次出现不一致） |
| 5 | 正反平衡 | 是否包含批评/局限/失败案例视角？还是只有正面论述？ |
| 6 | 形式合规 | 产出格式是否符合规范？（标题层级、引用标注、署名格式） |
| 7 | 校准抽检 | 从 PM 标了"确定"的论断中随机抽 2 处，对照搜索报告文件原文验证。自信度虚高 ≥2 处→标注"自评虚高"，打回重标 |
| 8 | 学习真实 | 是否包含"学习困惑"节？内容是真挣扎还是走过场？6b 费曼教学输出是否存在？ |

**7b 口试判定：** QA 根据 PM 闭卷回答判：✅真懂 / ⚠️部分懂 / ❌没懂。部分懂→针对性补学+重审该节。没懂→打回重学+重口试（最多 3 轮）。

**口试题 7 条设计原则（QA 出题时逐条对照）：**

| # | 原则 | 依据 |
|---|------|------|
| 1 | 不能照抄文档回答——必须用自己的话重组概念间关系 | 区分"读懂"和"会摘抄" |
| 2 | 必须连接 ≥2 个概念——问关系，不问孤立事实 | PhD viva 核心考综合能力 |
| 3 | 至少 1 题追问潜在矛盾或边界——"X 和 Y 在你的框架里冲突吗？" | Socratic irony 探盲区 |
| 4 | 格式：🟢基础题×1 → 🟡中档题×3 → 🔴挑战题×1（共 5 题，按难度递增） | 结构化格式+难度梯度 |
| 5 | QA 姿态：芒格式"我不太理解，你能解释一下吗"——对话不是审问 | 降低防御心理，暴露真实理解 |
| 6 | 难度梯度——至少 1 道挑战题预期 PM 答不完全，用来探知识边界。5 题全预期 ✅→题目偏易 | 区分"真懂"和"题简单" |
| 7 | 可外部校准——QA 出题时每题先写好"理想答案应覆盖的要点"（私存，不透露给 PM）。判定时对照核实 | 保证判定标准前后一致 |

判定规则：致命缺陷（结构缺失/自检虚假/内部自相矛盾/口试完全没懂）→ 打回。建议 → 标注不阻塞。存疑放行→条件必须明确化（"补 X 即过"）。PM 补完后 QA 在下一轮审查中先验条件是否满足再继续——条件验收不通过仍算打回。

**研究路径示例：**

CEO："搜索钱学森系统工程思维资料，学习达到研究生水平"

```
1. PM 判任务类型 → 研究类
2. PM 侦察 → 对钱学森系统工程有基础了解，跳过
3. PM 拆任务 → 四维度：原著/学术/应用/批评
4. QA 审拆法 → 3 轮打回后存疑放行（PM 角色边界需澄清）
5. [并行] 试点搜索 (1工程) → 通过 → 三路并行搜索 (3工程)
6. QA 审搜索结果 → 通过（建议补搜批评/局限）
7. PM 拼合 → 6a闭卷默写→6b教新手→6c输出学习文档(10章)→6d自检

**研究路径常见坑：**
- **把搜索当成学习** — 搜索阶段产出的是资料集，不是学习成果。学习转化是 PM 的工作，不在工程 spawn 范围内
- **让纯牛马做分析** — 纯牛马只搜不分析。交叉验证、概念提炼、对比分析需要判断力，超出纯牛马角色边界
- **跳过 QA 审拆法** — 即使没有代码，拆法（搜索维度、信息源分层）仍然需要 QA 审查
- **忘了定义信息源优先级** — 研究任务的信息源质量决定最终产出质量。必须在搜索前明确 L1-L4 分层和排除标准
- **跳口试（实验阶段）** — 前 3 次严格走口试，用于校准 QA 出题质量。轻量学习（≤2 概念）标"免口试"是少数例外
- **闭卷默写走过场** — Agent 摘要还在上下文里飘着→默写测的是复述能力，不是学习效果。PM 自问：我是从脑子里提取的，还是从刚才读到的 Agent 返回里复述的？诚实标"不确定"强于虚标"确定"。QA 抽检发现虚标→打回
- **搜索报告摘要≠原文** — PM 必须读至少 3 篇 L1 一手原文的关键论证段落。单纯重组 Agent 摘要不是学习
- **任务笔记完整度≠任务质量** — 格式完美的笔记+浅层执行 ≠ 好任务。PM 自问：笔记漂亮是因为任务做得好——还是模板设计得好？

## 三、PM 操作速查卡

PM小克接到 CEO 需求后的完整操作流程见**第一章步骤 0（0.1~0.5）**，角色 Prompt 模板见**第二章**。本章只列关键决策点和常见坑，不重复流程细节。

### 路由速查（详见第一章 0.1~0.5）

| 子步骤 | 做啥 | 一句话 |
|--------|------|--------|
| 0.1 | 查历史+了解现状 | 这事之前做过吗？涉及哪些文件？有什么约束？ |
| 0.2 | 理解需求 | CEO 要什么？怎样算完成？拿不准就问 |
| 0.2.5 | **判任务类型** | 代码类/研究类/混合类？类型决定后续所有路由 |
| 0.3 | 脑中快速拆 | 几步？要多路径吗？复杂度自评（研究类跳过文件预判） |
| 0.4 | 任务类型+三条件判路径 | 研究类→研究路径；代码类→≤4步？不需选方案？一个工程能独立干？ |
| 0.5 | 定路径+写笔记 | 路径决定写入任务笔记 → 跳转对应步骤 1 |

**关键约束：** 0.1~0.5 六段（含 0.2.5）都写完才能进路径。没写 = 后面决策缺依据。

### 路径入口速查

| 条件 | 路径 | 跳转 |
|------|------|------|
| 0.4.1 微路径判据全满足 | **微路径** | PM 自己做，0 spawn |
| 0.2.5=研究类 | **研究路径** | → 研究路径步骤 1 |
| 0.2.5=混合类 | 分拆——研究部分走研究路径，代码部分走三条件 | |
| ≤4步 + 单人 + 单路径 + 简单 | 短路径 | → 短路径步骤 1 |
| >4步 或 中等/复杂，方向确定 | 主流程·方向确定 | → 主流程步骤 1 |
| ≥2 个可行方向（不分步数） | 主流程·方向不确定 | → 多方案步骤 1 |
| 拿不准 | 默认主流程·方向确定 | 宁可多走，不冒走错的风险 |

### 升级条件速查

| 场景 | 触发条件 | 动作 |
|------|---------|------|
| 判错路径 | 执行到一半发现实际需要方案或多工程 | 叫停，已完成部分保留，升级到主流程 |
| QA 打回上限 | 方案或执行被同一 QA 打回 3 轮 | 升级 PM 裁决，不自动第 4 轮 |
| 架构超时 | 同一架构连续 2 次不出方案 | 换纯牛马代出方案 |
| 工程卡住 | 换 2 种方式仍不行 | 发求助给 PM，描述卡在哪+试过什么 |
| 工程冲突 | 两个工程逻辑打架 | 工程发冲突预警给 PM，PM 协调 |
| Agent 意见冲突 | 两 Agent 各执己见 | L1 自行辩论 → L2 PM/QA 裁决 → L3 CEO 拍板 |
| 架构方案雷同 | 多方案核心路径相同 | QA 打回重出，2 次雷同换人 |
| 财务严重超标 | 实际超估算 >50% | 财务立即报警 CEO |

### 常见坑

- **0.3 复杂度自评过于乐观** — 含改代码/改配置就标"复杂"，别骗自己是简单。判错升级比走错路径便宜
- **多方案锚点没拉开** — 三个架构给同一个锚点 = 三个雷同方案 = 白烧 token。锚点必须本质不同
- **财务估算等架构出完再跑** — 可以并行：财务基于 PM 拆分先估，方案出来微调
- **忘了派合规巡视** — 主流程必须有合规（步骤 6），不是可选项
- **短路径跳过了整体验证** — 短路径步骤 4 必须验证，只是跳过 QA 审方案和合规
- **任务笔记没写完就进路径** — 0.1~0.5 六段（含 0.2.5）缺一段，后面决策就缺一块依据
- **派工程不设边界** — 不写"禁止碰 X"，工程可能越界改红线区域
- **团队人数贪多** — 3-5 个队友最佳，超过 5 个协调成本指数增长。3 个聚焦的强过 5 个分散的
- **跨会话笔记丢失** — 任务笔记持久化在 `~/projects/晓组织/任务笔记/`（非 `/tmp/`），跨会话长任务不丢
- **产出格式不一致** — 各 agent 署名/结论标签/字段各不相同，拼合呈报时乱七八糟。PM spawn 前不检查格式，QA 审内容不审格式。格式规范见 agent 定义文件 + 拼合模板

## 四、QA/合规/财务执行脚本

### QA 方案审查脚本

```
1. 读方案全文
2. 逐项审 7 条清单
3. 发现致命缺陷 → 打回，写原因
4. 发现建议 → 标注，不阻塞
5. 拿不准 → 标注"存疑"，放行
6. 多方案时 → 加审雷同
7. 输出：[通过/打回] + 原因 + 建议 + 存疑。完整格式见 [.claude/agents/qa-reviewer.md](.claude/agents/qa-reviewer.md)
```

### QA 执行结果审查脚本

```
1. 读执行结果
2. 对照方案，逐项核实
3. 读纯牛马自验结论
4. 发现偏离 → 判断：方案问题还是执行问题
   - 方案问题 → 触发方案重审
   - 执行问题 → 打回修
5. 输出：[通过/打回] + 原因 + 建议。完整格式见 [.claude/agents/qa-reviewer.md](.claude/agents/qa-reviewer.md)
```

### 合规巡视脚本

```
1. 读方案摘要 + 执行结果
2. 对照 4 条审查
3. 分级（黄/橙/红）
4. 黄色 → 记录在署名结论
5. 橙色 → 记录 + 通知 PM
6. 红色 → 记录 + 通知 PM + CEO + 建议暂停
7. 输出：[级别] + 发现 + 建议。完整格式见 [.claude/agents/compliance.md](.claude/agents/compliance.md)
```

### 财务事前估算脚本

```
1. 读方案
2. 识别关键操作（搜索/读文件/写文件/调试等）
3. 按历史基线估算各角色 token
4. 标注明显的低效环节
5. 输出：估算表 + 备注。完整格式见 [.claude/agents/finance.md](.claude/agents/finance.md)
```

### 财务事后 ROI 脚本

```
1. 取实际 token 消耗
2. 对比事前估算
3. 计算偏差（%）
4. 逐角色 ROI 评估
5. 判断报警等级
6. 输出：[ROI报告] + [达标/异常/严重]。完整格式见 [.claude/agents/finance.md](.claude/agents/finance.md)
```

## 五、升级模板

### L1 — Agent 间协商

```
[Agent A] 和 [Agent B] 就 [议题] 存在分歧。

A 的立场：[简述 + 理由]
B 的立场：[简述 + 理由]

共同点：[列出]
分歧点：[列出]

请双方协商一轮（各陈述一次），找共识。不能达成 → 升级 L2。
```

### L2 — PM/QA 裁决

```
L1 协商未达成共识，升级至 PM [或 QA] 裁决。

议题：[描述]
A 立场：[简述]
B 立场：[简述]

请 PM 做裁决。QA 确认裁决不违反质量标准。

裁决后全员 disagree and commit——不再争论，全力执行裁决方向。
```

### L3 — CEO 拍板

```
L2 无法裁决，升级至 CEO(火影大人)。

议题：[描述]
A 立场：[原文]
B 立场：[原文]
PM 裁决意见：[如有]
QA 意见：[如有]

请火影大人拍板。
```
